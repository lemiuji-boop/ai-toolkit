# ТЗ-ФИНАЛ · Исполняемое ТЗ для Cursor: достройка до полноценной системы

> Кладётся как `docs/TZ-FINAL.md`. **Это главный и единственный план исполнения** — он консолидирует ТЗ-02v2, ТЗ-10v2 и ТЗ-v3 в одну последовательность задач. При противоречии с прежними документами действует этот. Прежние FR-ID сохраняются и указаны в задачах; новые требования — FR-080+.

---

## 0. Итоговое состояние («готово» = всё это правда одновременно)

1. Система работает на **любой** локальной модели (Ollama-тег или OpenAI-совместимый эндпойнт) — замена модели = изменение записи провайдера, без правок кода.
2. Внешние ИИ-сервисы подключаются **по API-ключам из админ-панели**, но конфиденциальные данные (сканы, обозначения, геометрия) на них физически не маршрутизируются — это гарантирует код, а не дисциплина.
3. В поставке **нет заглушек и неработающих модулей**: каждый путь кода либо реально работает, либо возвращает явную ошибку с причиной (503 + текст). Тестовые дубли живут только в `tests/`.
4. Сквозной сценарий работает на реальных данных: скан + STEP → каскад → каталог → расчёт → заполнение Excel с разметкой → исправление → корпус. Админ-панель: вход, мониторинг, провайдеры/ключи, данные, выгрузка.
5. CI зелёный; «гейт запрещённых паттернов» (раздел 3) пуст; восстановление из бэкапа отрепетировано.
6. Каталоги хранения данных настраиваются в админ-панели: полный анализ всех файлов и периодическая/ручная проверка новых наполняют базу; версии документов ведутся как «актуальная/архив» в разрезе комплектов изделий; извещения об изменении распознаются из PDF-сканов, связываются с версиями и сверяются.

## 1. Правила исполнения для Cursor (обязательны, дополняют .cursor/rules)

**Анти-зацикливание:**
- Одна задача T-NN за итерацию, строго по порядку раздела 6. Не начинать следующую, пока verify-команда текущей не зелёная.
- На починку упавшей проверки — максимум **2 попытки**. Третий провал → СТОП: записать состояние в `docs/BLOCKED.md` (задача, команда, вывод ошибки, гипотеза) и завершить итерацию. Не переписывать решение «с нуля» циклически.
- Запрещено «исправлять» проблему ослаблением теста, удалением проверки или расширением try/except. Тест меняется только если он противоречит этому ТЗ — с пометкой в `docs/trace.md`.

**Анти-галлюцинации:**
- Список зависимостей закрыт (раздел 5). Новая библиотека = СТОП + запись в `docs/OPEN_QUESTIONS.md`. Не выдумывать API библиотек: если сигнатура неизвестна — открыть установленный пакет в venv и посмотреть, либо СТОП.
- Не хватает данных для решения (имя хоста, формат, справочник) → не угадывать: взять значение из раздела 4 «Контракты»; если там нет — СТОП + OPEN_QUESTIONS.md.
- Любое значение, которого нет во входных данных, в ответ не подставлять (это правило системы и правило разработки одновременно).

**Анти-поломка:**
- Перед коммитом задачи: `ruff check . && mypy app && pytest -q` + verify-команда задачи. Всё зелёное — иначе СТОП-протокол.
- Менять только файлы из списка задачи. Контракты раздела 4 — неизменны; расширения только обратно совместимые.
- Каждая задача = отдельный коммит `T-NN: <суть>`. Откат всегда возможен на границу задачи.

## 2. Независимость от модели (обязательная архитектура)

```
app/services/llm/
  base.py        # Protocol LLMProvider: chat_vision(image,prompt,schema)->dict,
                 #                       chat_text(prompt,schema|None)->dict|str
  ollama.py      # OllamaProvider (локальный)
  openai_compat.py # OpenAICompatProvider (vLLM, llama.cpp-server, внешние API)
  router.py      # выбор провайдера по data_class задачи (см. раздел 2.2)
  registry.py    # реестр провайдеров из БД (таблица providers)
```

**2.1. Контракт ответа модели.** Каждый вызов передаёт JSON-схему ожидаемого ответа; ответ валидируется Pydantic. Невалидный JSON → один repair-повтор («верни строго по схеме, без пояснений»); снова невалидный → ошибка задачи с сохранением сырого ответа в лог задания (не выдумывать поля). Так система не зависит от «характера» конкретной модели.

**2.2. Маршрутизация по классу данных.** У каждой LLM-операции — метка: `confidential` (vision-извлечение, интегратор, всё с КД/обозначениями/геометрией) или `open` (подсказки по формулировкам замечаний нормоконтроля без данных изделия, справочные ответы). Маршрутизатор: `confidential` → только провайдеры `kind=local`; `open` → любой включённый по приоритету. Провайдер `kind=external` физически не получает confidential-вызов — проверяется тестом границы (FR-081).

## 3. Запрет заглушек в поставке

- Удаляются: деградация vision→stub, geometry→stub, фиктивные данные в продуктивных путях. Нет провайдера/cadquery/БД → **503/ошибка с причиной**, не подмена данными.
- Тестовые дубли — только в `tests/` через dependency injection (это не заглушки поставки).
- **Гейт в CI:** `grep -rnE "TODO|FIXME|NotImplementedError|source=\"stub\"|_stub\(" app/ addin/ admin/` обязан быть пустым; плюс CI-тест «нет провайдера → /api/jobs возвращает 503 с кодом причины, а не данные».

## 4. Контракты (неизменные; всё однозначно, чтобы нечего было угадывать)

- **Адреса:** сервер `https://matnorm.lan` (env `PUBLIC_BASE_URL`); статика надстройки и панель — nginx этого хоста; БД/MinIO наружу не публикуются.
- **API:** `POST /api/jobs` → 202 `{job_id}`; `GET /api/jobs/{id}` → `{status: queued|running|done|error, stage, result?, error?}`; SSE `GET /api/jobs/{id}/events`; `GET /api/catalog?q=&limit=&offset=`; `POST /api/corrections`; `POST /api/auth/login` → `{token}`; админ: `GET/POST/PUT /api/admin/providers`, `GET /api/admin/metrics`, `GET /api/admin/usage`, `GET /api/admin/export.xlsx`.
- **Таблица providers:** `id, name, kind('local'|'external'), base_url, api_key_encrypted(nullable), model, enabled(bool), priority(int), created_at`. Ключи шифруются Fernet ключом из env `SECRET_KEY`; в API и UI ключ маскируется `••••last4`; в логи не пишется (тест).
- **Каталоги и версии (миграция в T-23):** `product_sets(id, code unique, name)`; `watched_dirs(id, path, product_set_id FK, enabled, interval_minutes, last_scan_at)`; `part_versions(id, part_id FK, product_set_id FK, file_hash, source_path, doc_date, uploaded_at, status('actual'|'archive'), notice_id FK null, unique(part_id, product_set_id, file_hash))`; `notices(id, number unique, date, scan_object_key, affected jsonb)`; `scan_runs(id, dir_id, started_at, finished_at, files_seen, files_new, status)`.
- **Правило версий (детерминированное, без угадывания):** комплект изделия задаётся привязкой каталога к product_set (один каталог = один комплект). Новый файл: hash уже есть → пропуск; обозначение есть в комплекте → сравнение doc_date (из штампа/извещения; нет — mtime файла): более поздняя версия получает status=actual, прежняя actual того же (part, product_set) → archive. То же обозначение в **другом** комплекте — независимая actual-версия (новая ревизия другого комплекта). Первая версия любого документа — actual.
- **Извещения:** распознавание PDF-скана извещения — confidential-вызов (router→local); извлекаются номер, дата, перечень затронутых обозначений. Сверка: actual-версия без связанного извещения → флаг `VERSION_WITHOUT_NOTICE`; извещение с обозначением без новой версии в базе → флаг `NOTICE_WITHOUT_VERSION`. Флаги — данные, не блокировка.
- **API каталогов:** `GET/POST/PUT/DELETE /api/admin/dirs`; `POST /api/admin/dirs/{id}/scan?mode=full|new` → 202 + scan_id; `GET /api/admin/scans/{id}` + SSE прогресса; `GET /api/admin/versions?part=&set=&status=`; `POST /api/admin/notices` (загрузка PDF) и `GET /api/admin/notices`.
- **Классы данных:** enum `DataClass = confidential|open`; точка маршрутизации одна — `router.choose(data_class)`.
- **Схема ответа vision (как в каркасе):** `ExtractResult` + `zones, candidates, rules_version, model_version` (опциональные поля, обратная совместимость).
- **Форма листа Excel:** колонки как в `excel_format.HEADERS`; обозначения — текстовый формат; числа — числовой формат с локалью; сомнительное: заливка `#FAEEDA` (низкая уверенность) / `#FAD7C0` (конфликт с 3D) + примечание с причиной.
- **rules.json:** обязательное поле `version` (int, монотонно); загрузка без version → ошибка старта.

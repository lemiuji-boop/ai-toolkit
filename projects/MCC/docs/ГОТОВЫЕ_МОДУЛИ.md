# ГОТОВЫЕ_МОДУЛИ · Инструкция встраивания для Cursor

> Кладётся как `docs/ГОТОВЫЕ_МОДУЛИ.md`. Дополняет `docs/TZ-FINAL.md` и протокол `.cursor/rules/40-execution.mdc` — **не отменяет их**: порядок задач, стоп-протокол и контракты §4 действуют без изменений. Этот документ сообщает Cursor, какие части задач уже реализованы и протестированы человеком-архитектором, чтобы их **встраивать, а не разрабатывать заново**.

---

## 1. Главное правило

Перечисленные ниже модули — **готовый, протестированный код** (23 теста зелёные). Для них действует режим «встроить/заменить»:

- **НЕ переписывать** тело модулей и их тесты. Разрешено только: импортировать, вызывать, добавлять тонкие адаптеры рядом (новые файлы), расширять обратно совместимо (новые функции/поля со значениями по умолчанию).
- Если задача T-NN из TZ-FINAL требует функциональность, уже покрытую готовым модулем, — задача сводится к **подключению** модуля в указанные точки и дописыванию недостающей обвязки.
- Конфликт между формулировкой задачи и готовым модулем → модуль прав (он соответствует контрактам §4); расхождение фиксируется в `docs/OPEN_QUESTIONS.md`, модуль не правится.
- Перед началом каждой задачи: открыть этот документ, найти задачу в таблице §2, выполнить «сверку до» (см. §4), и только потом писать код.

## 4. Протокол сверки (выполняется в каждой задаче)

**Сверка ДО начала задачи:**
1. Прочитать формулировку T-NN в `docs/TZ-FINAL.md` (файлы, verify-команда).
2. Найти T-NN в таблице §2 этого документа. Есть готовый модуль → режим «встроить»; нет → обычная разработка.
3. Прогнать `pytest -q` — базовая линия зелёная до изменений.

**Сверка ВО ВРЕМЯ:**
- Любая попытка изменить файл из §2 или его тест → СТОП, перечитать §1.
- Сигнатуры неизвестны → открыть сам модуль (он в репозитории), не выдумывать.
- Контракты (коды ошибок, имена таблиц, формат маски ключа) → только TZ-FINAL §4.

**Сверка ПОСЛЕ:**
1. Verify-команда задачи + `ruff check . && mypy app && pytest -q` — всё зелёное.
2. Тесты готовых модулей не изменены: `git diff --stat tests/ app/services/llm app/services/versioning.py app/services/rules_registry.py app/services/jobs_runner.py app/core/crypto.py app/db/models.py` — пусто (или только новые файлы).
3. Строка в `docs/trace.md`; коммит `T-NN: ...`. Провал дважды → стоп-протокол 40-execution.

## 5. Что осознанно НЕ сделано за Cursor (чтобы не плодить мёртвый код)

- Alembic-миграции (генерируются от моделей в среде Cursor — так они совпадут с его версией Alembic);
- DBJobStore и все роуты (зависят от session.py, который появится в ШАГЕ B);

Эти места — обычные задачи TZ-FINAL: контракты заданы, готовые модули дают опорные интерфейсы.
